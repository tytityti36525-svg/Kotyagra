# Серверний анти-чіт для КотяГри (Cloud Functions)

Ці функції працюють на сервері Firebase і **виправляють накручені дані**,
навіть якщо гравець змінює їх напряму в базі. Клієнт переписувати не треба —
сервер просто підрівнює неможливі значення назад.

## Що всередині
- `onPlayerWrite` — на кожен запис гравця перевіряє монети/сердечка/рівень/тижневий
  рахунок і виправляє, якщо вони завеликі або стрибнули неможливо різко.
- `onMailCreate` — обрізає задовгі листи й завищені суми подарунків.
- `weeklySweep` — раз на годину чистить рейтинг від явних читерів.

## ⚠️ Потрібен план Blaze
Cloud Functions працюють лише на тарифі **Blaze** (оплата за використання).
Для маленької гри це майже безкоштовно (є щедрий безкоштовний ліміт), але
картку прив'язати треба. Увімкнути: Firebase Console → ⚙️ → Usage and billing → Modify plan → Blaze.

## Як задеплоїти (один раз)

1. Встанови Node.js 20+ і Firebase CLI:
   ```
   npm install -g firebase-tools
   firebase login
   ```

2. У корені проєкту (там, де лежить ця папка `functions/`) ініціалізуй, якщо ще нема `firebase.json`:
   ```
   firebase init functions
   ```
   - Обери свій проєкт **KotyaGra**.
   - Мова: **JavaScript**. Коли спитає перезаписати `index.js`/`package.json` — **відмовся (N)**, щоб не стерти ці файли.

3. Постав залежності й задеплой:
   ```
   cd functions
   npm install
   cd ..
   firebase deploy --only functions
   ```

4. Готово. У Console → Functions побачиш `onPlayerWrite`, `onMailCreate`, `weeklySweep`.

## Налаштування меж
У `functions/index.js` зверху є об'єкт `CAP` і функція `weekCap` — підлаштуй
числа під свою економіку (макс. монети, межа тижневого рахунку для рівня тощо).

## Що це НЕ робить
Це захист «постфактум» (сервер виправляє після запису). Щоб гравець **взагалі
не міг** писати монети напряму, всі нарахування треба перенести у callable-функції
(`onCall`) і заборонити запис цих полів правилами. Це великий рефакторинг клієнта —
скажи, якщо захочеш зробити повну серверну економіку, підготую покроково.

## Лог підозрілих
Виправлення пишуться в колекцію `cheatLog` — можеш переглядати, хто намагався накрутити.

## 🔐 Серверна економіка (перенесення з клієнта)

Це **поетапний** процес. Зроблено перший і другий етапи:

**Етап 1 — серверний контроль значень (вже працює через `onPlayerWrite`):**
Сервер на кожен запис перевіряє монети/сердечка/метеликів/рівень і **відкочує накручені значення**. Ліміти підтягнуто під реальну гру (макс. приріст за один запис трохи вищий за найбільшу чесну нагороду — експедиція ~2800🪙), тож чесна гра не страждає, а редагування бази вручну — марне.

**Етап 2 — авторитетні нагороди на сервері (`claimWeeklyReward`):**
Приз за тижневий турнір тепер **рахується й нараховується на сервері** за реальним рейтингом. Підробити суму чи ранг на клієнті неможливо. Захист від повторного отримання — поле `weekClaimedKey`.
Клієнт викликає цю функцію автоматично; поки функції не задеплоєні — працює старий клієнтський розрахунок (відкат), тож гра не ламається. **Після деплою** приз стає захищеним сам собою.

**Що далі (за бажанням, наступні етапи):**
- Перенести так само **щоденний бонус** і **колесо фортуни** (callable-функції `claimDaily`, `spinWheel`), щоб їх не можна було отримувати повторно через редагування дати.
- Перенести **реферальні нагороди** на сервер.
- Після переносу всіх ключових нарахувань — заборонити в `firestore.rules` пряму зміну `coins`/`hearts` клієнтом (лишити лише серверу). Це фінальний етап повної серверної економіки.

⚠️ Кожен етап потребує деплою функцій (`firebase deploy --only functions`) і тесту. Раджу впроваджувати по одному, перевіряючи гру після кожного.
